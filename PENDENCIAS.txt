SEÇÃO 1: DÉBITOS TÉCNICOS E ARQUITETURA (PRIORIDADE CRÍTICA)
Estes itens colocam a estabilidade, segurança ou escalabilidade do sistema em risco.
=============================================================================

[01] Implementação de Testes Automatizados (Backend)
     - Detalhe: A pasta `src/test` está vazia. O sistema não possui garantia de qualidade automática.
     - Ação: Criar testes unitários com JUnit 5 e Mockito para os principais Services (`CicloService`, `RegistroEstudoService`).
     - Meta: Cobrir cenários críticos de cálculo de horas e validação de regras.

[02] Otimização de Cálculos no Banco de Dados (Performance)
     - Detalhe: `DashboardService` carrega listas inteiras em memória para somar segundos via Java Stream. Isso não escala.
     - Ação: Refatorar consultas JPQL para usar agregações (SUM, AVG, GROUP BY) diretamente no PostgreSQL.

[03] Externalização de Segredos (Segurança)
     - Detalhe: Senhas de banco e Secret JWT estão expostos no `application.properties` e no código.
     - Ação: Configurar leitura via Variáveis de Ambiente (`System.getenv`) e criar arquivo `.env` (não versionado) para desenvolvimento local.

[04] Observabilidade e Logs (Infraestrutura)
     - Detalhe: Ausência de logs estruturados. Erros em produção serão invisíveis.
     - Ação: Adicionar Logback/Slf4j nos Services para registrar fluxos críticos (ex: "Ciclo X criado", "Erro ao processar registro Y").

[05] Dockerização para Produção (Infraestrutura)
     - Detalhe: Existe apenas o Compose do banco. Falta o container da aplicação Java.
     - Ação: Criar `Dockerfile` otimizado (Multi-stage build) para gerar a imagem leve do .jar da API.

SEÇÃO 2: UX E FRONT-END (MELHORIA DE EXPERIÊNCIA)
Melhorias necessárias para o sistema deixar de parecer um protótipo.
=============================================================================

[06] Sistema de Notificações (Toasts)
     - Detalhe: O sistema usa `window.alert()` para erros e sucessos, o que trava o navegador e é visualmente pobre.
     - Ação: Implementar Context API + Componente de Toast para feedback não bloqueante (Sucesso/Erro/Info).

[07] Feedback de Carregamento (Skeleton Screens)
     - Detalhe: O uso de texto simples "Carregando..." causa layout shift (pulo de tela).
     - Ação: Criar componentes de Skeleton (esqueleto cinza pulsante) para Tabelas e Cards do Dashboard.

[08] Tratamento de Erros no Formulário
     - Detalhe: Mensagens de erro do backend nem sempre são exibidas de forma amigável no input correspondente.
     - Ação: Padronizar o consumo de exceções da API no front para destacar campos inválidos em vermelho.
